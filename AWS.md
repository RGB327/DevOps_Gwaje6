# AWS

---
## 퍼블릭 클라우드
> AWS, Google cloud 등의 클라우드 서비스 제공 업체에서 인터넷을 통해 서버, 스토리지, 네트워크 등의 컴퓨팅 자원을 제공하는 모델

**장점**
- **확장성&유연성**
    - 필요할 때 필요한 만큼 유동적으로 리소스를 조절할 수 있음
- **비용 효율성**
    - 서버를 설치하는 데 드는 막대한 초기 비용을 무시할 수 있고 사용한 만큼만 비용을 지불하는 구조로 효율적인 비용 운용이 가능함
- **유지보수&관리의 편의성**
    - 물리적인 서버의 관리는 클라우드 제공사가 맡기 때문에 유지보수와 관리 면에서 신경을 쓰지 않아도 됨
- **고가용성&재해복구**
    - 전 세계 여러 데이터 센터에 데이터를 분산 저장해 특정 지역에 재해가 발생해도 서비스가 중단되지 않음

**단점**
- **보안 및 데이터 프라이버시**
    - 다른 공간에서 데이터를 보관하기 때문에 유출위험이 높음
- **비용의 불확실성**
    - 사용한 만큼 비용을 내지만 사용량이 급증하면 예상치 못한 지출이 발생할 수 있음
- **특정 기업 의존성(Vendor lock-in)**
    - 특정 제공업체의 기술에 종속되어 부조리한 상황이 생겨도 옮기기 쉽지 않음

*IaaS , PaaS , SaaS 등의 서비스들이 여기에 속함*
## 프라이빗 클라우드
> 특정 기업이나 조직이 전용으로 사용하는 독립된 클라우드 환경

**장점**
- **보안성**
    - 자체적인 보안 정책과 네트워크를 가지고 있어 민감 정보 보호에 유리
- **예측 가능한 비용**
    - 관리비, 유지비 등의 고정적 지출 외엔 사용량으로 인한 지출 들이 없어 안정적인 비용 예측이 가능하다
- **맞춤형 구성**
    - 특정 하드웨어나 소프트웨어 등 좀 더 세부적으로 요구에 맞게 커스터마이징 할 수 있음

**단점**
대부분 퍼블릭 클라우드의 장점에 상반됨(초기 비용, 유지보수 등등)


## 하이브리드 클라우드
> 민감한 데이터는 프라이빗에 일반 업무는 퍼블릭으로 분리해 사용하는 방식

**특징**
퍼블릭 클라우드와 프라이빗 클라우드의 단점을 보완한 클라우드 컴퓨팅 방식임

**보완점**
1. 퍼블릭 클라우드 방식의 보안 문제점을 완화할 수 있음
2. 프라이빗 클라우드 방식의 재해 대처 부분을 보완할 수 있음
3. 프라이빗 클라우드의 유연성이 부족한 부분을 매꿀 수 있음
**알맞은 자원 분배를 통해 더 효율적이고 안전한 운용 가능**


# IAM

> Identity and Access Management의 약자로 조직의 IT 리소스에 대한 접근 권한을 안전하게 인증 및 인가하는 보안 시스템

## 사용하는 이유
여러 사용자와 서비스가 있는 환경에서 보안을 유지하면서도 필요한 자원에만 접근하도록 통제하기 위해 사용

## 구성 요소

#### 사용자 (User)

- 실제 사람이나 애플리케이션을 의미하는 IAM의 기본 단위
- 각각 고유한 자격 증명(비밀번호, Access Key 등)을 가짐
- 예: 개발자 계정, 관리자 계정, 서버 애플리케이션

---

#### 그룹 (Group)

- 여러 사용자를 묶어 관리하기 위한 단위
- 동일한 권한이 필요한 사용자들을 하나의 그룹으로 관리
- 그룹에 정책을 부여하면 속한 모든 사용자에게 자동 적용됨
- 예: 개발팀, 운영팀, 관리자 그룹

---

#### 역할 (Role)

- 특정 권한을 가진 **임시 자격** 개념
- 사용자나 서비스가 필요할 때 역할을 **Assume(임시로 획득)** 해서 사용
- 비밀번호 없이도 권한을 위임할 수 있음
- 주로 서비스 간 권한 부여나 외부 접근 제어에 사용
- 예: EC2가 S3에 접근할 때 사용하는 Role

---

#### 정책 (Policy)

- 권한을 정의하는 JSON 형태의 규칙 문서
- 누가(Who) 어떤 리소스에 대해 무엇을(What) 어떤 조건에서(Condition) 할 수 있는지를 명시
- Allow(허용) / Deny(거부)로 구성
- 사용자, 그룹, 역할에 연결하여 권한을 부여함

**예시 구조**

```json
{
  "Effect": "Allow",
  "Action": "s3:GetObject",
  "Resource": "*"
}
```

# Auto Scaling
> 애플리케이션의 부하(트래픽)에 따라 컴퓨팅 자원의 양을 자동으로 조절해주는 서비스

#### 오토 스케일링 방식
- **스케일 아웃(Scale out)**: 서버의 대수를 늘리는 방식
- **스케일 인(Scale-in)**: 부하가 줄어들면 서버의 대수를 다시 줄이는 방식
- **스케일 업(Scale-up)**: 서버 자체의 사양을 높이는 방식
- **스케일 다운(Scale-down)**: 서버 사양을 다시 내리는 방식

*스케일 업/다운 방식은 변경한 사양을 적용하는 시간이 필요해 서비스가 중단될 수 있어 잘 사용하지 않음*

---

#### 필요한 이유
- **비용 절감**: 사용자가 적은 시간대에는 사용하는 자원의 양을 줄여 비용 절감 가능
- **가용성 확보**: 갑작스러운 트래픽 폭주에도 대응할 수 있어 서비스가 중단되지 않음
- **내결함성**: 특정 서버에 장애가 발생하면 이를 감지해 새 인스턴스를 자동 생성해 대체(Health check 기반)

---

#### 작동방식
AWS에서는 ASG(Auto Scaling Group)을 통해 관리

- **Auto Scaling Group**: EC2 인스턴스 집합을 관리하는 논리적 그룹
- 최소(Min), 최대(Max), 원하는 수(Desired)를 설정하여 인스턴스 개수 유지
- 항상 Desired(유지하려는 상태)를 유지하도록 자동 조정



# ELB
> Elastic Load Balancing의 약자로 서버로 들어오는 트래픽을 분산시켜 부하를 줄여주는 서비스

#### 주요 역할
- **트래픽 분산**: 들어오는 클라이언트의 요청을 여러 대의 서버(EC2, 컨테이너 등)로 골고루 나눠줌
- **상태 확인 (Health Check)**: 연결된 서버들이 정상인지 주기적으로 확인, 장애가 있으면 트래픽 전달을 즉시 중단
- **자동 등록/제거**: AutoScaling으로 자동 생성된 인스턴스를 자동 연결하고 그 반대의 역할도 함
---
#### 종류
- **ALB(Application Load Balancer)**
    - 7계층 특화로, URL, 경로 기반의 라우팅 기능을 함
- **NLB(Network Load Balancer)**
    - 4계층 특화로, 매우 빠르고 대량의 트래픽 처리에 최적화 되어있음

*주로 Auto Scaling으로 생성된 여러 인스턴스에 트래픽을 분산하고 확장성을 높이는 역할을 맡음*

# RDS
> Relational Database Service의 약자로 AWS에서 지원하는 관리형 관계형 데이터베이스 서비스이다

#### 사용하는 이유
- **편의성**
    - 서버 사양,OS, DB엔진 설치 등을 간편하게 설정할 수 있고 AWS가 자동으로 관리하기 때문에 운영 부담이 감소함
- **자동 백업 및 자동 복구**
    - 지정된 시간에 자동 백업을 수행하며 특정 시점으로 데이터를 돌리는 스냅샷 기능을 제공함
- **고가용성**
    - 기존 서버에 장애 발생 시 다른 AZ의 데이터베이스 서버(복제본)를 사용하여 (Failover) 서비스를 유지할 수 있음

#### 단점
- **비용**: 편의성을 제공하는 서비스이기 때문에 EC2에 DB를 구축하는 것 보다 비싸다
- **OS및 DB제어 제한**: AWS에서 관리하는 서비스라 서버에 직접적인 접근(SSH)이 제한되고 버전 선택이나 플러그인 설치 등에서 제약이 생길 수 있음
---
**지원하는 데이터베이스 종류**

- **MySQL** - 가장 많이 사용되는 오픈소스 관계형 데이터베이스, 웹 서비스에 적합
- **PostgreSQL**- 표준 SQL을 잘 지원하며 확장성과 기능이 뛰어남
- **Amazon Aurora** - AWS에서 제공하는 고성능 관계형 데이터베이스로 MySQL/PostgreSQL과 호환되며 높은 성능과 자동 확장 기능을 제공
- **MariaDB** - MySQL과 호환되면서 성능과 기능이 개선된 오픈소스 DB
- **Oracle** - 대기업 환경에서 많이 사용하는 상용 데이터베이스로 안정성과 기능이 뛰어남
- **SQL Server** - Microsoft에서 제공하는 DB로 .NET 환경과의 호환성이 뛰어남



# CloudWatch
> CloudWatch란 AWS리소스와 애플리케이션의 성능, 로그, 이벤트를 실시간으로 모니터링하고 시각화하는 완관리형 서비스이다.

#### 사용하는 이유
- **시스템 상태 확인**: CPU, 메모리, 네트워크 등의 상태를 실시간으로 확인
- **문제 조기 감지**: 이상 징후를 빠르게 발견하여 장애 예방
- **자동화 트리거**: 특정 조건 만족 시 Auto Scaling 등과 연동 가능

### 기능
- **Metrics**
    - 리소스의 상태를 수치로 제공함
    - 기본 제공 + 커스텀 메트릭 생성 가능
    - **예**: EC2 CPU 사용률 , RDS DBConnections
    - 일정 주기로 수집 (기본 5분 / 상세 모니터링 1분)
    통계 (평균, 최대, 최소 등) 제공

- **Logs**
    - 애플리케이션 및 시스템 로그 수집
    - 로그 그룹 / 로그 스트림 구조
    - **활용**:
    에러 추적
    특정 문자열 검색 (필터링)
    - **추가 기능**:
    Logs Insights: SQL처럼 로그 분석 가능
    로그 보존 기간 설정 가능

- **Alarm**
    - 특정 조건을 설정하고 기준 초과 시 알림 발생
    - **예**:
    CPU 사용률 > 80% -> 알림
    - **동작 방식**:
    Metric 감시
    조건 충족 시 상태 변경 (OK -> ALARM)
    SNS, Lambda 등으로 알림/자동 실행
    - **상태**: OK / ALARM / INSUFFICIENT_DATA

- **Dashboard**
    - 여러 리소스 상태를 한 화면에서 시각적으로 확인
    - 커스텀 구성 가능
    - 운영자가 전체 시스템 상태를 빠르게 파악 가능


- **Event**
    - 특정 이벤트 발생 시 자동 작업 수행
    - **활용 예**:
     EC2 상태 변경 -> 자동 스크립트 실행
     특정 시간 -> 배치 작업 실행
    - *현재는 Amazon EventBridge로 확장됨*
